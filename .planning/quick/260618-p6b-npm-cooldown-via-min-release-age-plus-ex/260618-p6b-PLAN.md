<objective>
Lean into npm's native `--min-release-age` cooldown for npm packages, make the install
script policy explicit, and enforce a registry-only dependency source posture. Delete the
redundant hand-rolled npm-version selection in `resolve-versions.sh` (npm 11.16 does both
version selection and transitive-tree pinning natively in one flag). govulncheck stays in
`resolve-versions.sh` because it is a Go-proxy package, not an npm package.

Purpose: Less custom code, native npm cooldown for npm packages, explicit/auditable install
script policy, registry-only sources — with the existing fail-closed `verify-pins.sh` as the
absolute backstop (unchanged).

Output: Modified Dockerfile, resolve-versions.sh, build-and-lock.sh, CLAUDE.md.

**Architectural consequence:** npm package versions become known only post-build (read from
the extracted `versions-npm.json`) instead of pre-resolved by `resolve-versions.sh`.
`build-and-lock.sh` must read them from that snapshot when assembling `versions.lock`.

**Key safety property (from approved plan):** `verify-pins.sh` already re-checks every
resolved package (top-level AND transitive, from `versions-npm.json`) against the ABSOLUTE
`COOLDOWN_DATE` in `versions.lock` and fails closed. So even though `--min-release-age` is
relative to build wall-clock, the verifier is the absolute backstop. The relative window
(≥N×24h) is slightly stricter than the calendar-day cutoff at the boundary, so it will not
spuriously trip the verifier.

**Explicitly UNCHANGED:** `scripts/verify-pins.sh` (reads versions.lock + versions-npm.json,
re-checks all against COOLDOWN_DATE, fails closed — behavior identical once build-and-lock.sh
writes the post-build-resolved versions into the lock) and `rebuild.sh` (already passes
`--cooldown-days`/`--tag`/`--build-date` to build-and-lock.sh; COOLDOWN_DAYS threads through
unchanged). Do NOT modify either file.
</objective>

layer is invalidated on every cooldown-date change — not relying solely on the line-6 anchor.

postinstall in 1.4.0; `prepare` does not run for registry installs; real setup is the explicit
`gsd-core --claude --global`). claude-code gets `--allow-scripts @anthropic-ai/claude-code`

transitive-dep scripts).

flags (all|none|root) defaulting to `all` (permissive). Pass `=none` on both installs to enforce
registry-only. Direct deps of gsd-core 1.4.0 and claude-code 2.1.169 are all registry semver
ranges (no git+/github:/url/file: specs). A blocked transitive source fails closed via
verify-pins.sh WR-02 __MISSING__ sentinel.
