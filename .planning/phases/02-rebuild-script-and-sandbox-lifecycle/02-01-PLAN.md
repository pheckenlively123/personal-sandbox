<objective>
Add build-date tagging and a podman-inspectable cooldown/provenance label to the
image, satisfying BLD-03. This is the first vertical slice of Phase 2: after this
plan, a single `build-and-lock.sh --tag claude-sandbox:<date> --build-date <date>`
invocation produces a date-tagged image whose `podman inspect` Labels carry the
cooldown date and build metadata. The downstream `rebuild.sh` (Plan 02) hands the
date-pinned ref to `openshell sandbox create` and relies on this flag existing.

Purpose: BLD-03 / ROADMAP success criterion #2 — "The image produced is tagged with
the build date and carries the cooldown date as an image label, visible via
`podman inspect`." Per locked decisions D-03 (date tag) and D-04 (LABEL-via-ARG so
provenance travels with the image regardless of build entry point).

Output: Extended `Dockerfile` (ARG BUILD_DATE + 5 LABEL lines) and extended
`scripts/build-and-lock.sh` (--build-date flag + BUILD_DATE passthrough).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator CLI args → build-and-lock.sh | `--build-date` value is operator-supplied and flows into a `podman build --build-arg`; untrusted-format input crosses here |
| registry responses → resolver → build-arg | COOLDOWN_DATE/version strings (already validated by the Phase 1 allowlist in lines 71-95) feed the new LABEL lines |
| build ARGs → Dockerfile LABEL | ARG values are interpolated into image metadata baked into the image |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering/Injection | `--build-date` arg → `--build-arg BUILD_DATE=` | mitigate | Allowlist-validate BUILD_DATE against `^[0-9]{4}-[0-9]{2}-[0-9]{2}$` immediately after arg-parse; reject before any podman invocation (Task 2). No eval. |
| T-02-02 | Tampering | resolver registry values → LABEL lines | accept | Already mitigated upstream by the Phase 1 KEY=VALUE allowlist parse (build-and-lock.sh lines 71-95); LABELs only consume already-validated ARGs. No new exposure. |
| T-02-03 | Information disclosure | provenance labels baked into image | accept | Labels carry only build/cooldown dates and pinned versions — no secrets, no PII. This is the intended BLD-03 provenance surface. |
</threat_model>
